Question Link : https://leetcode.com/problems/range-sum-query-mutable/description/

*   We need to support two operations efficiently:

        1. update(index, val)
        2. sumRange(left, right)

*   Brute force approaches fail:

        normal array:
            update -> O(1)
            query  -> O(n)

        prefix sum:
            update -> O(n)
            query  -> O(1)

*   Segment Tree balances both operations efficiently:

        update -> O(log n)
        query  -> O(log n)

*   Core idea:
        Divide array into smaller segments recursively
        and store precomputed information for every segment.

*   Each Segment Tree node represents:
        a subarray range.

*   Example:

        nums = [1,3,5,7]

        root:
            [0...3]

        left child:
            [0...1]

        right child:
            [2...3]

*   Tree construction continues recursively
        until every leaf node stores exactly one array element.

*   Leaf nodes represent:
        single elements.

*   Internal nodes represent:
        merged result of child ranges.

*   Here merge operation is:
        SUM

*   Example:

        node [0...3]
        stores:

            sum([0...1]) + sum([2...3])

*   Query operation works using 3 overlap cases:

        1. Complete Overlap
        2. No Overlap
        3. Partial Overlap

*   Complete Overlap:
        current segment lies completely inside query range.

    So:
        directly reuse stored answer.

    This is the main optimization of Segment Tree.

*   No Overlap:
        current segment contributes nothing.

    Since operation is SUM,
        identity element becomes 0.

*   Partial Overlap:
        recursively query left and right children
        and merge their results.

*   During update:
        only one root-to-leaf path changes
        because only one array index is modified.

*   After updating leaf node,
        recompute all parent nodes while recursion backtracks.

*   Segment Tree height is O(log n),
        because every level halves the segment size.

*   Therefore:
        both update and query become logarithmic.

*   Segment Tree is more powerful than Fenwick Tree because:
        it supports more complex range operations,
        lazy propagation,
        range updates,
        min/max/gcd queries etc.

*   Array representation of Segment Tree:

        node i:
            left child  -> 2*i + 1
            right child -> 2*i + 2

*   We allocate size 4*n because:
        recursive tree may not form perfect complete binary tree.

*   Time Complexity:

        build:
            O(n)

        update:
            O(log n)

        query:
            O(log n)

*   Space Complexity:
        O(4n)
